# check-prompt-for-opus

Opus 4.7 に与えるプロンプトを **総合的にレビューして整える** ための skill。これ 1 本で「適切かどうかの判定 → 改善案の提示 → ファイル書き出し」までを完結させることをゴールとする。

チェックは次の 3 軸で行う:

- **A. 主要構造要素**: Goal(目的) / Constraints(制約) / Acceptance criteria(完了条件) の必須 3 要素 + 関連ファイルの場所 (推奨)
- **B. 基本文章品質**: 曖昧語、矛盾、主語/対象の欠落、過剰冗長、構造の崩れ、明らかな誤字
- **C. Opus 4.7 適切性**: 出力形式の明示、構造化、コンテキストの why、例示の活用、エージェント的指示、不要な persona、古い CoT 技法、他モデル前提の記述

各指摘には **重大度** を付ける (色 + 形 + 塗りつぶし系の単純な記号で表現):

| マーク | 意味 | 修正の必要性 |
|--------|------|-------------|
| 🛑 重要 | 指示が成立しない / Opus が誤った成果物を出しやすい | 必ず直す |
| 🟨 中 | 現状でも動くが、改善すると安定性・効率が上がる | 直すことを推奨 |
| 🟢 軽微 | スタイル・好みレベル。直さなくてもよい | 任意 |

色は赤 / 黄 / 緑、形は八角形 / 四角 / 丸でそれぞれ別。後述するサマリ表示の `✅ / ⚠️ / ❌` とは別の系統の記号を使い、混同しないようにしている。

## なぜ Opus 4.7 向けに整えるのか

Opus 4.7 のような高性能 LLM は雑な指示でもそれなりに動くため、書き手は「通じている」と錯覚しやすい。実際には、必須要素の欠落・曖昧語・モデル特性に合わない書き方が紛れていると、再現性が落ちたりスコープが暴走したりする。**プロンプトを 1 度整えるだけで、以降の試行コストと事故率が大きく下がる**。

## 3 軸の判定基準

### A. 主要構造要素

| 要素 | 区分 | 何を書くか | 欠けるとどうなるか |
|------|------|-----------|---------------------|
| **Goal(目的)** | 必須 | 何を達成したいか。最終アウトプットの姿 | 手段ばかりで目的を見失い、的外れな解に行きやすい |
| **Constraints(制約)** | 必須 | 守るべきルール、禁止事項、対象範囲、性能・運用・互換性などの縛り | スコープが膨らみ、無駄な改修や危険な変更が混ざる |
| **Acceptance criteria(完了条件)** | 必須 | 何が満たされていれば「終わり」とみなせるか。検証方法 | 完了を自己判断できず、過剰に手を広げるか早すぎる打ち切りになる |
| **関連ファイルの場所** | 推奨 | 変更・参照対象のファイルパス、ディレクトリ、設定ファイル、関連コードの場所 | Opus が当て推量でファイルを探し回り関係ないファイルに手をつける恐れがある；またはファイルの場所をユーザーに確認する往復が発生しトークン効率が低下する |

判定のコツ:

- 見出し / ラベル / 箇条書き / 明示的な文末表現で、読み手が即座にその要素として認識できることを「明確に記載」とみなす。
- 文章中に紛れていてラベル等が無い場合は「明確に記載されていない」扱い。
- 表記ゆれは同義扱いで OK (Goal ≒ ゴール ≒ 目的 ≒ 狙い、Constraints ≒ 制約 ≒ 制限 ≒ 守るべきこと、Acceptance criteria ≒ 完了条件 ≒ Done の定義 ≒ 成功条件)。
- 必須 3 要素の欠落は原則 🛑 重要 (Goal だけは内容次第で 🟨 中もありうる)。
- **関連ファイルの場所** の欠落は 🟨 中 (コード変更・ファイル参照を伴うタスクでのみ指摘する。プロンプトが概念的な質問や設計相談など特定ファイルを必要としない場合は指摘不要)。

### B. 基本文章品質

以下に該当する箇所があれば指摘する:

- **曖昧語** 🟨 〜 🛑: 「いい感じに」「適切に」「うまく」「なるべく」「だいたい」「ちゃんと」「キレイに」「最適に」など、解釈幅が大きい語が **判定基準として使われている** とき。具体例や数値・条件に置き換えるよう促す。
  - 例: 「適切なエラーハンドリングをする」→ 何が「適切」かが不明 → 「想定する例外と挙動 (リトライ / フォールバック / 中止) を表で示す」など具体化を提案。
- **矛盾・両立不能な指示** 🛑: 同時に成立しない指示が混在している場合。
  - 例: 「既存ロジックは一切変えずに大幅にリファクタする」「下位互換を保ったまま API シグネチャを刷新する」など。
- **主語 / 対象の欠落** 🛑 〜 🟨: 「修正する」「対応する」など動詞だけで主語または対象が不明な箇所。
- **過剰冗長 / 前置きの長さ** 🟨 〜 🟢: 本題に入るまでに無関係な背景説明が多すぎる、同じことを 3 回以上言い換えている、など。
- **構造の崩れ** 🟨: 1 つの段落に独立した複数の要件が混在し、見出し / 箇条書きで分けられていない場合。
- **明らかな誤字脱字 / 不自然な表現** 🟢: 意味が通るが読みにくくなっているもの。

「文体の好み」「丁寧さ」「絵文字を入れたほうがいい」のような **書き手の趣味の領域には踏み込まない**。

### C. Opus 4.7 適切性

Anthropic の Claude 4 系プロンプトベストプラクティスと Opus 4.7 の特性 (高い推論力 / コード生成 / 1M トークンコンテキスト / エージェント的タスク対応) を踏まえた観点:

- **出力形式の明示** 🟨 〜 🛑: 求めるフォーマット (Markdown / JSON / コード / 表 など)、言語 (日本語 / 英語)、想定の長さ・粒度が書かれているか。あえて柔軟性を残したい場合は「自由に」と明示しているか。
- **構造化** 🟨: 長い指示が地の文で書かれているとき、見出し / 箇条書き / XML タグ (`<context>`, `<requirements>`, `<output_format>` など) で分けられているか。Claude は XML タグの境界を強く尊重する。
- **コンテキスト (why) の提示** 🟨: 「何を」だけでなく「なぜ」(背景・目的・利用者・優先順位の理由) を書いているか。why があると Claude 4 系は判断の質が安定する。
- **例示 (few-shot)** 🟨 〜 🛑: 形式や品質を揃えたい / 微妙なニュアンスがある場合に、入出力の具体例が 1 〜 3 件あるか。フォーマット指定だけで具体例が無いと崩れやすい。
- **エージェント的タスクの整備** 🛑: ツール使用 / コード変更 / 自律的実行を伴う場合、次が書かれているか:
  - 「問題が完全に解決するまで継続する」(早すぎる打ち切り防止)
  - 使ってよい / 禁止する tool・コマンド
  - 並列実行してよい範囲、独立タスクの並列化を許可するか
  - 不確実なときは推測せず確認・調査するスタンス
  - 完了の検証方法 (テスト実行 / ログ確認 など)
- **不要な persona / role play** 🟨: 「あなたは天才エンジニアです」「あなたは 20 年経験のシニア …」のようなペルソナ付けは Opus 4.7 ではほぼ効果が無く、ノイズになる。タスクと無関係なら削除を提案。役割を絞り込みたい場合は能力ベース (「コードレビューに集中して」) で書く。
- **古い CoT 技法** 🟨: "Let's think step by step" / 「順を追って考えてください」だけを書く形式は Opus 4.x では暗黙にやるため冗長。明示的に思考を促すなら `<thinking>` タグや interleaved thinking、または「主要な選択の理由を最初に短く述べてから実装」のように **何を考えてほしいか** まで指定する。
- **モデル / 環境前提の不整合** 🛑: 「あなたは GPT-4 です」「あなたは Claude 3 Opus です」のような他モデル前提の記述、廃止された tool 名、明らかに古いコンテキストウィンドウ前提の指示など。
- **ネガティブ命令過多** 🟨: 「〜しないで」ばかりで、肯定形 (何をしてほしいか) が薄い場合。Claude は肯定形指示の方が安定する。
- **過剰な逐語固定** 🟢: 出力を全部テンプレで固定するより、構造 (見出し / セクション順) だけ指定したほうが品質が出るケースで、逐語固定しているとき。

## ワークフロー

以下の 5 ステップを順に実行する。

### Step 1. 対象プロンプトを確定する

チェック対象が曖昧なまま進めない。

- **既に提示されている場合** (引数 / ファイル参照 / 直前メッセージなど): それをそのまま対象にする。
- **未提示の場合**: ユーザーに「どのプロンプトを対象にしますか? 本文を貼ってもらうか、ファイルパスを教えてください」と確認する。
- **チャット履歴から起こせそうな場合**: 直近のやりとりから Opus 4.7 用プロンプトに仕立てられそうでも、**勝手に作らない**。「これまでのやりとりからプロンプトを起こしてもよいですか?」と必ず確認してから進める。

### Step 2. 3 軸で総合チェック

A → B → C の順で、対象プロンプトを精読しながら指摘候補を内部的に列挙する。各指摘について次を控える:

- どの軸 (A/B/C) のどの観点か
- 該当箇所 (引用または位置)
- 重大度 (🛑 / 🟨 / 🟢)
- なぜ問題か (一文)

ノイズになる指摘は出さない。具体的には:

- 軽微 (🟢) は **2 〜 3 件まで** に抑える。それ以上見つかっても代表のみ。
- 同じ問題の繰り返しは 1 件にまとめる。
- 書き手の趣味領域 (文体・絵文字・敬体/常体) には踏み込まない。

### Step 3. チェック結果を構造化して伝える

次のテンプレに沿って結果を返す。**必ずこの 3 ブロック構成** で出す。

```
## チェックサマリ
- A. 主要構造要素: ✅ / ⚠️ / ❌
- B. 文章品質: ✅ / ⚠️ / ❌
- C. Opus 4.7 適切性: ✅ / ⚠️ / ❌

## 指摘 (重大度順)
🛑 [軸X] 観点名: 該当箇所「…」 — 一文の理由
🟨 [軸X] 観点名: 該当箇所「…」 — 一文の理由
🟢 [軸X] 観点名: …

## 総合判定
- このまま投げて良い / 軽微な修正で投げて良い / 修正してから投げるべき のいずれか + 一文サマリ
```

凡例 (記号は **2 系統** あるので混同しないこと):

- **サマリ行** (軸ごとの集約結果) — `✅` 指摘なし / `⚠️` 中・軽微のみ (重要なし) / `❌` 重要を含む
- **指摘行の重大度マーク** (個別の指摘ごと) — `🛑` 重要 / `🟨` 中 / `🟢` 軽微

すべての軸が ✅ の場合は、指摘ブロックを省略して **「総合判定: このまま投げて問題ありません」だけを返して終了** する。余計な改善提案を上乗せしない。

### Step 4. 修正案を提示する (指摘がある場合のみ)

🛑 と 🟨 については、**個別に具体的な書き換え案** を出す。🟢 はまとめて短く触れるだけで良い。

修正案の出し方:

1. 元プロンプトのどの部分をどう書き換えるかを **before / after** で示す。
2. 不足要素 (Goal / Constraints / Acceptance criteria) を補完するときは、本文から類推できる根拠 + 具体案をセットで示す。本文から類推できない要素は「ユーザー側で補ってほしい」と明示する。無理に推測で埋めない。
3. 修正案は **Opus 4.7 への指示として実用に耐える具体性** を持たせる。曖昧語 (「適切に」「いい感じに」) を修正案の中で再生産しない。
4. 元プロンプトの体裁と意図 (敬体/常体、英日、見出し有無など) は尊重する。スタイル変更は提案しない。

修正案を提示したら、最後に必ず「**修正版をファイルに書き出しますか? 出すならファイル名 (例: `prompts/your-task.md`) を教えてください**」と確認する。

### Step 5. ファイル書き出し (ユーザー希望時のみ)

ユーザーがファイル名を指定した場合のみ、修正済みプロンプト全文をそのファイルに書き出す。

書き出しルール:

- 確認なしには絶対に書き出さない。
- 出力形式は元プロンプトの体裁を尊重しつつ、Goal / Constraints / Acceptance criteria は **見出しまたは明示的なラベル** で分かるようにする。
- 修正案以外の本文は元のまま残す (大幅リライトしない)。
- 書き出し後は、「書き出しました: `path`」と一言で報告して終了する。

ユーザーが「ファイルにはしなくていい」と答えたらそのまま終了する。

## 例

### 例 1: 全軸 ✅

入力プロンプト:
```
## Goal
社内ナレッジ DB から「障害対応」タグの記事を月次でサマリし、Slack #ops-weekly に投稿する。

## Constraints
- 記事 URL は社外秘なので公開チャネルには貼らない
- 投稿は週次バッチではなく月初 1 営業日の 09:00 JST 固定
- 既存の slack-bot ライブラリを利用し、追加依存は入れない

## Acceptance criteria
- 月初 1 営業日 09:00 に投稿が 1 件される
- 件名・本文・記事タイトル一覧 (URL なし) が含まれる
- ステージング環境で 2 ヶ月分のドライラン結果に問題が無い

## 出力形式
実装は Python 3.12 + 既存の `tools/slackbot/` を編集する。差分のみを示す。
```

出力:
> ## チェックサマリ
> - A. 主要構造要素: ✅
> - B. 文章品質: ✅
> - C. Opus 4.7 適切性: ✅
>
> ## 総合判定
> このまま Opus 4.7 に投げて問題ありません。

### 例 2: A 軸欠落 + 補完

入力プロンプト:
```
社内ナレッジ DB から「障害対応」タグの記事を月次でサマリして Slack に投稿する仕組みを作って。
URL は社外秘だから載せないで。月初 1 営業日の 09:00 JST 固定で動かす想定。
既存の slack-bot を使ってほしい。
```

出力:
> ## チェックサマリ
> - A. 主要構造要素: ⚠️ (Constraints と Acceptance criteria がラベル付きで明示されていない)
> - B. 文章品質: ✅
> - C. Opus 4.7 適切性: ⚠️ (出力形式の指定なし)
>
> ## 指摘
> 🟨 [A] Constraints のラベルなし: 「URL は載せないで」「09:00 JST 固定」「既存 slack-bot 利用」が地の文に混在 — Opus が制約として認識しづらい
> 🟨 [A] Acceptance criteria 未記載: 完了判定の文言がない — 自己判定できず過剰に手を広げる恐れ
> 🟨 [C] 出力形式未指定: 実装の納品物 (差分パッチ / 全ファイル / コマンド) が不明 — 期待と違うフォーマットになる可能性
>
> ## 総合判定
> 修正してから投げるのを推奨します。
>
> ## 修正案
> 以下のセクションを追記すれば 3 軸とも ✅ になります。
>
> - **Constraints** (本文から類推):
>   - 投稿に記事 URL は含めない
>   - 投稿時刻は月初 1 営業日 09:00 JST 固定
>   - 既存 slack-bot ライブラリを使い、追加依存は入れない
> - **Acceptance criteria** (本文には未記載だが Goal から導出):
>   - 月初 1 営業日 09:00 JST に Slack 投稿が 1 件作成される
>   - 投稿に「障害対応」タグの記事タイトル一覧が含まれ、URL は含まない
>   - ステージングでのドライランが連続 2 ヶ月成功する
> - **出力形式**: 既存 `tools/slackbot/` 配下を編集する diff を示す。新規ファイルが必要ならファイルパスと全文を併記。
>
> 修正版をファイルに書き出しますか? 出すならファイル名 (例: `prompts/ops-weekly-summary.md`) を教えてください。

### 例 3: B / C 軸での指摘

入力プロンプト:
```
あなたは天才フロントエンドエンジニアです。Let's think step by step。
このプロジェクトのパフォーマンスをいい感じに改善してください。既存の挙動は絶対に一切変えないでください。
バンドルサイズも大幅に削ってください。
```

出力:
> ## チェックサマリ
> - A. 主要構造要素: ❌ (Goal が曖昧、Acceptance criteria なし、Constraints は内部矛盾)
> - B. 文章品質: ❌ (曖昧語 + 矛盾)
> - C. Opus 4.7 適切性: ⚠️ (不要な persona、古い CoT 技法、出力形式なし)
>
> ## 指摘
> 🛑 [B] 矛盾: 「挙動は一切変えず」+「大幅に削る」 — 厳密に挙動を保ったまま大幅縮小は不可能。優先順位や許容範囲を決める必要あり
> 🛑 [B] 曖昧語: 「いい感じに」「大幅に」 — 改善目標値や測定方法が無いと完了判定できない
> 🛑 [A] Acceptance criteria なし: 改善後の合格ラインが書かれていない
> 🟨 [C] 不要な persona: 「あなたは天才フロントエンドエンジニアです」 — Opus 4.7 では効果薄。能力ベース (例: 「バンドル最適化のレビュー観点で」) のほうが効く
> 🟨 [C] 古い CoT 技法: "Let's think step by step" 単体は Opus 4.x では冗長。考えてほしい観点を具体化したほうがよい
> 🟨 [C] 出力形式未指定: 報告書 / PR diff / 実装手順のどれが欲しいか不明
>
> ## 総合判定
> 修正してから投げるべき。現状では Opus が完了判定も最適化方針も決められません。
>
> ## 修正案
> - **Goal**: 「初回ロード時間 (FCP) を計測し、◯◯ 秒 → ◯◯ 秒に短縮する。並行して JS バンドルを ◯◯ KB 以下にする」のように測れる目標に置き換える。
> - **Constraints の矛盾解消**: 「ユーザー観測の挙動 (描画結果・操作フロー) は変えないが、内部実装 (依存ライブラリ・コード分割・遅延ロード) は変更可」のように **挙動の何を保ち、何を変えていいか** を分ける。
> - **Acceptance criteria**: 「Lighthouse Performance スコア 90 以上」「主要 3 ページの FCP が ◯◯ 秒未満」「既存 e2e テストが全 pass」など、計測可能な合格ラインを書く。
> - **persona / CoT 削除**: 「あなたは天才…」と "Let's think step by step" は削除。代わりに「最適化を入れる前に、現状のバンドル分析結果と最大ボトルネック上位 3 件を提示してから実装に入る」のように考える観点を具体化。
> - **出力形式**: 「(1) 計測レポート (2) 改善方針の選択肢と推奨案 (3) 実装の差分 を順に出してください」のように構造を指定。

## やらないこと

- Opus 4.7 以外のモデル向け最適化 (GPT / Gemini / 旧 Claude 系の癖に合わせた書き換えはしない)
- プロンプトが扱う **ドメイン知識そのもの** の妥当性検証 (例: 「この医療判断の手順は正しいか」「この SQL は要件を満たすか」など、プロンプト中身の専門領域の正誤判断はしない)
- 書き手の文体・語調・敬体常体の好みへの介入
- 「もっと丁寧に」「もっと褒めて」のような感情・態度の調整
- 確認なしのチャット履歴からのプロンプト生成
- 確認なしのファイル書き出し
- 元プロンプトの全面リライト (あくまで指摘箇所の最小修正にとどめる)
